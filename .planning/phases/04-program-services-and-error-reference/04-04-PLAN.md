Frontmatter: title "Error Handling Patterns", description "Code examples for handling the most common Protochain error scenarios — retryable errors, submission certainty, and program execution failures."

Opening: Protochain returns structured errors that tell you whether to retry, rebuild, or check on-chain state. The patterns below cover the most common scenarios.

Reference callout: "For the full list of error codes with retryability classifications and remediation actions, see the [Error Reference](/api-reference/errors)."

---

## Pattern 1: Checking if an error is retryable

Show how to inspect the error code and decide whether to retry. The errors.mdx classifies codes as Permanent, Temporary, or Indeterminate.

Include CodeGroup Go/Rust/TypeScript showing:
- Catching a SubmitTransaction error
- Extracting the TransactionErrorCode
- Temporary errors (INSUFFICIENT_RESOURCES, RATE_LIMITED, NETWORK_ERROR) → retry with backoff
- Permanent errors (INVALID_TRANSACTION, INVALID_SIGNATURE, etc.) → don't retry, return error to caller
- Indeterminate errors → check certainty (see Pattern 2)

---

## Pattern 2: Handling indeterminate submission (TransactionSubmissionCertainty)

Indeterminate errors are the trickiest case — you don't know if the transaction was sent. The structured_error.certainty field tells you how to proceed.

Include CodeGroup Go/Rust/TypeScript showing:
- CERTAINTY_SENT: the transaction was sent, use GetTransaction or MonitorTransaction to poll for result
- CERTAINTY_NOT_SENT: the transaction was not sent, safe to retry or resubmit
- CERTAINTY_UNKNOWN: cannot determine; use GetTransaction to check; if not found after reasonable wait, resubmit

---

## Pattern 3: Handling program execution failures

PROGRAM_ERROR means the on-chain program rejected the instruction. The error itself is generic — use GetTransaction to get the detailed meta_error_message from the on-chain logs.

Include CodeGroup Go/Rust/TypeScript showing:
- Detecting TRANSACTION_ERROR_CODE_PROGRAM_ERROR
- Calling GetTransaction with the signature to retrieve meta_error_message
- Logging or surfacing the program-specific error message

---

## Pattern 4: Handling expired blockhash

Compiled transactions expire after ~150 slots. BLOCKHASH_EXPIRED means you need to recompile (not just re-sign).

Include CodeGroup Go/Rust/TypeScript showing:
- Detecting TRANSACTION_ERROR_CODE_BLOCKHASH_EXPIRED (or CheckIfTransactionIsExpired returning expired)
- Recompiling the transaction: call CompileTransaction again (fetches a fresh blockhash), then SignTransaction, then SubmitTransaction

Show this as a brief code snippet with a comment: "// Recompile to get a fresh blockhash — do not just retry with the same compiled transaction."
