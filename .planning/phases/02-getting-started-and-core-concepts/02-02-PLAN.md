  TRANSACTION_STATE_COMPILED = 2;    // Instructions compiled to data, read-only
  TRANSACTION_STATE_PARTIALLY_SIGNED = 3; // Some signatures added
  TRANSACTION_STATE_FULLY_SIGNED = 4;    // All signatures present, ready for submission
}

CompileTransaction: DRAFT → COMPILED
SignTransaction: COMPILED → PARTIALLY_SIGNED or FULLY_SIGNED

EstimateTransaction: COMPILED state (non-mutating)
SimulateTransaction: COMPILED state (non-mutating)
CheckIfTransactionIsExpired: checks FULLY_SIGNED before submit

}

**Page structure:**

1. **Opening paragraph (2-3 sentences):** Every transaction in Protochain moves through a defined sequence of states before it can be submitted to Solana. The API enforces this sequence — you cannot sign a transaction that isn't compiled, or submit one that isn't fully signed. Understanding these states is essential for working with the Transaction Service.

2. **ASCII state machine diagram (per D-07 — no Mermaid):**

```
DRAFT
  │
  │  CompileTransaction
  ▼
COMPILED
  │
  │  SignTransaction
  ▼
PARTIALLY_SIGNED ──── SignTransaction (remaining signers) ────┐
                                                              │
                                                              ▼
                                                        FULLY_SIGNED
                                                              │
                                                              │  SubmitTransaction
                                                              ▼
                                                         (on-chain)
```

Use a code block with no language tag to render this as monospace. Make the ASCII art clear and readable.

3. **"States" section:** Short prose description of each state:
   - **DRAFT** — The transaction is being assembled. Instructions can be added or removed. The transaction has no compiled representation yet.
   - **COMPILED** — Instructions have been serialized into the Solana transaction wire format. The transaction now has a `recent_blockhash` and can be estimated, simulated, or signed. No new instructions can be added.
   - **PARTIALLY_SIGNED** — One or more signers have added their signatures, but not all required signers have signed. Multi-signer transactions pass through this state.
   - **FULLY_SIGNED** — All required signers have signed. The transaction is ready to submit. Check expiry before submitting if time has passed since compilation.

4. **"A typical transaction" walkthrough section (per D-04 — practical guide format):**
   Walk through the common single-signer path as numbered steps (use `<Steps>` component):
   - Step 1: Build instructions using the System Program or Token Program service. Each instruction call returns a `SolanaInstruction` — it does not execute anything yet.
   - Step 2: Call `CompileTransaction` with the draft transaction (containing the instructions), a fee payer address, and optionally a `recent_blockhash`. The transaction moves to COMPILED state.
   - Step 3 (optional): Call `EstimateTransaction` to get the compute units and fee. Call `SimulateTransaction` to dry-run the transaction before committing.
   - Step 4: Call `SignTransaction` with the compiled transaction and your signing key. For a single-signer transaction, this moves directly to FULLY_SIGNED.
   - Step 5: Call `CheckIfTransactionIsExpired` if significant time has passed since compilation. A transaction expires when its `recent_blockhash` ages out (~2 minutes / ~150 slots on mainnet).
   - Step 6: Call `SubmitTransaction`. This returns immediately — it does not wait for confirmation. Use `MonitorTransaction` to track the transaction's progress on-chain.

5. **"Blockhash expiry" note:** A `<Note>` callout — compiled transactions have a limited validity window tied to their `recent_blockhash`. If you don't submit within approximately 150 slots (~2 minutes on mainnet), the transaction expires. You must recompile with a fresh blockhash. Use `CheckIfTransactionIsExpired` before submitting if time has passed.

6. **Footer links:** Link to the Transaction Service API reference (forward reference, page will exist after Phase 3): "See the [Transaction Service](/api-reference/transaction/overview) reference for method details."

**Frontmatter:**
```
title: "Instructions and Transactions"
description: "How instructions from the System Program and Token Program services compose into Protochain transactions."
```

**Page structure:**

1. **Opening paragraph:** In Protochain, transactions are built by composing instructions. The System Program Service and Token Program Service act as instruction factories — each method call returns a `SolanaInstruction` (or a list of them) that you add to a draft transaction. The instructions are not executed when you call these methods; execution happens when you compile, sign, and submit the transaction via the Transaction Service.

2. **"What is a SolanaInstruction?" section:** Explain the three fields that matter:
   - `program_id` — The on-chain program that will execute this instruction (e.g., the System Program or Token Program address).
   - `accounts` — The accounts this instruction reads from or writes to, each with `is_signer` and `is_writable` flags that the Solana runtime enforces.
   - `data` — The serialized parameters the program will receive when the instruction executes.

   Note: You don't construct `SolanaInstruction` manually. You call a System Program or Token Program method and receive the instruction back.

3. **"Building a transaction from instructions" section:** Walk through the pattern conceptually (no code per D-05):
   - You have a goal: transfer SOL from account A to account B.
   - You call `SystemProgram.Transfer(from, to, amount)` — this returns a `SolanaInstruction`.
   - You put that instruction in a `Transaction` message in DRAFT state (`transaction.instructions = [instruction]`).
   - You compile and submit the transaction. The System Program executes when the transaction lands on-chain.

   This is the "instruction builder pattern" — the program services build instructions; the Transaction Service executes them.

4. **"Multi-instruction transactions" section (short):** A single transaction can contain multiple instructions. They execute atomically — either all succeed or all fail. This is how complex operations work: create a token account and mint tokens in one transaction, or create an account and initialize it in one step.

5. **"Why this design?" section (short):** This separation lets you combine instructions from multiple programs in one transaction, estimate fees before committing, and simulate execution before submitting. It mirrors how native Solana transactions work.

6. **Footer links:** Link forward to System Program Service and Token Program Service API references: "See the [System Program Service](/api-reference/system-program/overview) and [Token Program Service](/api-reference/token-program/overview) references for the full list of instruction builder methods."
