**Frontmatter:**

title: "Commitment Levels"
description: "How Solana's three confirmation stages map to the CommitmentLevel enum — and when to use each."
```

**Page structure:**

1. **Opening paragraph:** Solana validators process transactions through three stages of confirmation, each representing a higher degree of cluster consensus. Protochain exposes this as the `CommitmentLevel` enum, which appears on many request messages. Choosing the wrong level is a common source of race conditions — use this page to pick the right one for your use case.

2. **"How Solana confirmation works" section (brief):** Explain the three stages in plain terms:
   - A transaction is first processed by the current leader and included in a block. At this point it exists on one node but the rest of the cluster hasn't confirmed the block yet.
   - A supermajority of validators (2/3+) vote on the block, confirming it. The block is now highly unlikely to be rolled back.
   - The block becomes finalized (rooted) once the cluster has built enough subsequent blocks on top of it. A finalized block cannot be reversed.

3. **"CommitmentLevel values" section:** Use a simple comparison table (not a list) — this is the discretion area in CONTEXT.md. Table columns: Level | When the data is available | Risk of rollback | Typical latency | Use when.

   Rows:
   - PROCESSED | As soon as a block is produced | Low, but possible | ~400ms | You need the absolute lowest read latency and can tolerate the small risk of a dropped block (e.g., non-critical UI updates, building read-only tooling)
   - CONFIRMED | After supermajority vote | Very low | ~800ms–1.5s | Most operations — the default. Account balance reads, transaction status checks during interactive flows.
   - FINALIZED | After block is rooted | None — irreversible | ~13s (32+ slots) | Before crediting a user account, confirming a deposit, or any action where rollback would be harmful.
   - UNSPECIFIED | Service default applies | — | — | When you have no specific requirement. The service uses CONFIRMED unless documented otherwise.

4. **"Which level should I use?" section:** Short prose guidance:
   - Default to `CONFIRMED` — it's the right choice for most API calls.
   - Use `FINALIZED` for actions where a rollback would cause real harm: confirming an SOL deposit before crediting a user, verifying a payment, or any irreversible off-chain action.
   - Use `PROCESSED` only for non-critical reads where you prioritize speed over certainty — never for fund movements.
   - Pass `UNSPECIFIED` (or omit the field) when you want the service's documented default.

5. **Footer:** Link to the enum reference: "For the full enum reference, see [CommitmentLevel](/api-reference/shared-types#commitmentlevel) in Shared Types."

**Frontmatter:**
```
title: "Token Programs"
description: "SPL Token and Token-2022 are two incompatible Solana token programs. This page explains when to use each."
```

**Page structure:**

1. **Opening paragraph:** Solana has two token programs: the original SPL Token program (`TOKEN_PROGRAM_LEGACY`) and the newer Token Extensions program (`TOKEN_PROGRAM_2022`). They are not interchangeable — a token mint is permanently tied to one program, and all operations on that token must use the matching program. Mixing them causes failures that can be difficult to diagnose.

2. **`<Warning>` callout:** `TOKEN_PROGRAM_UNSPECIFIED` is not a valid value. If you pass it to any Protochain method, the service returns an error. Always specify either `TOKEN_PROGRAM_LEGACY` or `TOKEN_PROGRAM_2022`.

3. **"SPL Token (LEGACY)" section:** Explain:
   - The original Solana token standard, deployed in 2020. The vast majority of existing tokens (USDC, SOL wrapped tokens, most DeFi tokens) use this program.
   - Program address: `TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA`
   - Stable, widely supported, maximum ecosystem compatibility.
   - Does not support extensions (transfer fees, interest-bearing tokens, confidential transfers, etc.).
   - **Use when:** you are working with existing SPL tokens, or you need maximum compatibility with wallets, DEXes, and other Solana infrastructure.

4. **"Token-2022 (2022)" section:** Explain:
   - The Token Extensions program (also called Token-2022), deployed in 2022 and enabled on mainnet in 2023. Built to extend the token standard without breaking the existing program.
   - Program address: `TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb`
   - Supports extensions: transfer fees, interest-bearing tokens, confidential transfers, non-transferable tokens, metadata pointer, and more.
   - Less universal support in older wallets and protocols than LEGACY.
   - **Use when:** you are creating a new token and need extension features, or you are explicitly building with Token-2022 infrastructure.

5. **"Incompatibility is permanent" section (short):** Once a mint is created with one program, it cannot be migrated to the other. All token accounts (holding accounts / ATAs) for that mint must also use the same program. The `token_program` field must be consistent across all operations involving a given mint.

6. **"Which should I choose?" section:** Short decision guide:
   - Already working with an existing token? Use whatever program that token was created with. Check the mint's on-chain program address if unsure.
   - Creating a new token that needs transfer fees or other extensions? Use `TOKEN_PROGRAM_2022`.
   - Creating a new token with no special requirements? Use `TOKEN_PROGRAM_LEGACY` for maximum compatibility.

7. **Footer:** Link to the enum reference and Token Program Service: "For the full enum reference, see [TokenProgram](/api-reference/shared-types#tokenprogram) in Shared Types. For token creation and minting methods, see the [Token Program Service](/api-reference/token-program/overview)."
