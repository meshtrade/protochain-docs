**Opening paragraph:** Frame this as "your first real transaction" — connects naturally from the quickstart. One short paragraph. No subheading.

**Section: "What you'll build"**
One sentence: transferring lamports from one account to another using the System Program's Transfer instruction, compiled and submitted via Transaction Service.

**Section: "Prerequisites"**
Brief bullet list:
- Two funded Solana accounts (payer and recipient). See [Quickstart](/guides/quickstart) to generate and fund accounts on devnet.
- A Protochain connection (see [Connecting to Protochain](/guides/connecting)).

**Section: "Build the transfer instruction"**
Prose: The [System Program's Transfer method](/api-reference/system-program/transfer) returns a `SolanaInstruction` — it does not execute on-chain. The instruction is assembled into a transaction in the next step. See [Instructions & Transactions](/concepts/instructions-and-transactions) for the pattern.

CodeGroup (Go, Rust, TypeScript) showing only the Transfer call:
- Go: `systemClient.Transfer(ctx, &system_v1.TransferRequest{From: senderPubKey, To: recipientPubKey, Lamports: 1_000_000_000})`
- Rust: `system_client.transfer(tonic::Request::new(TransferRequest { from: sender_pub_key, to: recipient_pub_key, lamports: 1_000_000_000 })).await?`
- TypeScript: `systemClient.transfer(req, callback)` using setFrom/setTo/setLamports

**Section: "Compile the transaction"**
Prose: Add the instruction to a `Transaction` object in DRAFT state, then call [CompileTransaction](/api-reference/transaction/compile-transaction). Protochain fetches a recent blockhash automatically if `recent_blockhash` is left empty.

CodeGroup showing CompileTransaction call with the instruction wrapped in a Transaction. Use the Transaction proto structure (instructions field is a repeated SolanaInstruction). Show fee_payer being set to the sender's public key.

**Section: "Sign the transaction"**
Prose: Call [SignTransaction](/api-reference/transaction/sign-transaction) with the compiled transaction and the sender's private key. The transaction advances to FULLY_SIGNED state.

CodeGroup showing SignTransaction with `private_keys` signing method. Keep snippet minimal — one call, show the state advance in a comment.

**Section: "Submit the transaction"**
Prose: Call [SubmitTransaction](/api-reference/transaction/submit-transaction). The call returns immediately with a `signature` — it does not wait for confirmation.

Add a Note callout: `SUBMISSION_RESULT_SUBMITTED` means the transaction was accepted for broadcast, not confirmed. Use the signature with MonitorTransaction next.

CodeGroup showing SubmitTransaction, checking `submission_result`, and printing the signature.

**Section: "Monitor the result"**
Prose: Pass the signature from SubmitTransaction to [MonitorTransaction](/api-reference/transaction/monitor-transaction). The stream closes when the transaction reaches a terminal state (FINALIZED, FAILED, DROPPED, or TIMEOUT).

CodeGroup showing the MonitorTransaction stream loop with full terminal-state handling (all 4 terminal cases). Adapt from the existing code on the MonitorTransaction reference page — same pattern.

**Section: "Next steps"**
Short bullet list:
- Create a token mint: [Create a Token](/guides/create-token)
- Monitor multiple transactions: [Monitor Transactions guide](/guides/monitor-transaction)
- Full Transfer reference: [System Program Transfer](/api-reference/system-program/transfer)
