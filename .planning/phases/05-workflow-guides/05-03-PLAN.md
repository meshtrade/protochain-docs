**Opening paragraph:** Two sentences. MonitorTransaction is a server-streaming RPC — the server pushes status updates until the transaction reaches a terminal state. This guide covers production usage: understanding the full lifecycle, handling every terminal outcome, and reconnecting reliably when the stream drops.

Reference the [MonitorTransaction reference page](/api-reference/transaction/monitor-transaction) for request/response field documentation.

**Section: "Stream lifecycle"**
Prose: When you call MonitorTransaction, the server begins watching the transaction's on-chain status and emits a new response each time it advances. The stream closes automatically when the transaction reaches a terminal state.

Include the TransactionStatus table (same data as the reference page, reprinted here for convenience as this is a self-contained guide):

| Status | Terminal? | Meaning |
|--------|-----------|---------|
| `TRANSACTION_STATUS_RECEIVED` | No | Received by a validator |
| `TRANSACTION_STATUS_PROCESSED` | No | Processed (Processed commitment) |
| `TRANSACTION_STATUS_CONFIRMED` | No | Confirmed (Confirmed commitment) |
| `TRANSACTION_STATUS_FINALIZED` | **Yes** | Finalized — transaction succeeded |
| `TRANSACTION_STATUS_FAILED` | **Yes** | On-chain execution failed |
| `TRANSACTION_STATUS_DROPPED` | **Yes** | Not processed; blockhash may still be valid |
| `TRANSACTION_STATUS_TIMEOUT` | **Yes** | Monitoring window expired |

Add a Note callout distinguishing TIMEOUT from FAILED: TIMEOUT means the monitoring window expired, not that the transaction failed. DROPPED means the transaction was not processed by any validator.

**Section: "Handling terminal states"**
Prose: Each terminal state requires a different response.

Use a definition-style layout — 4 sub-sections (H3 or bold inline):

**FINALIZED** — Transaction succeeded and is permanent. No further action needed.

**FAILED** — Transaction executed on-chain but failed (e.g., insufficient balance, program error). Check `error_message` for details. See the [Error Reference](/api-reference/errors) for error codes. Do NOT resubmit the same transaction — the failure is deterministic.

**DROPPED** — The transaction was not processed. Check whether the blockhash has expired using [CheckIfTransactionIsExpired](/api-reference/transaction/check-if-expired). If not expired: resubmit with [SubmitTransaction](/api-reference/transaction/submit-transaction). If expired: recompile with [CompileTransaction](/api-reference/transaction/compile-transaction) to get a fresh blockhash.

**TIMEOUT** — The monitoring window expired (default: 60 seconds). The transaction may still be processing. Two options:
1. Call [GetTransaction](/api-reference/transaction/get-transaction) to check current status.
2. Start a new MonitorTransaction call with the same signature.

No code in this section — prose only. The code is in the reconnection section.

**Section: "Production pattern: reconnect with backoff"**
Prose (2-3 sentences): In production, two things can go wrong beyond terminal states: the gRPC stream itself can drop (network error), or TIMEOUT can fire before the transaction finalizes. The pattern below handles both: it retries on stream errors with exponential backoff and restarts monitoring on TIMEOUT.

CodeGroup (Go, Rust, TypeScript) — this is the centrepiece of the guide.

Each language example should show a `monitorWithRetry` function (or equivalent) that:
1. Takes a signature, a transaction client, a max-attempts count, and a base backoff duration
2. Loops up to max attempts
3. On each attempt, opens a MonitorTransaction stream
4. Reads messages in the stream loop
5. On FINALIZED: returns success
6. On FAILED: returns error immediately (no retry — deterministic failure)
7. On DROPPED: returns a "dropped" error (caller decides whether to resubmit)
8. On TIMEOUT: increments attempt counter, applies exponential backoff (base × 2^attempt), continues loop
9. On stream/network error: logs the error, increments attempt counter, applies exponential backoff, continues loop
10. After max attempts: returns a "max retries exceeded" error

Go implementation notes:
- Use `time.Sleep` for backoff
- Backoff formula: `backoff := baseDuration * time.Duration(1<<attempt)` — cap at e.g. 30 seconds
- Distinguish `io.EOF` (clean stream close) from actual errors

Rust implementation notes:
- Use `tokio::time::sleep` for backoff
- `Duration::from_secs(base_secs * 2u64.pow(attempt))` for exponential backoff
- Return `Result<(), Box<dyn std::error::Error>>`

TypeScript implementation notes:
- Use a Promise-based wrapper around the stream event emitter
- Backoff: `Math.min(baseSecs * 2 ** attempt, maxSecs) * 1000` milliseconds
- `await new Promise(resolve => setTimeout(resolve, backoffMs))` for the sleep

Keep each language's function to ~40-50 lines. Focus on correctness, not brevity.
